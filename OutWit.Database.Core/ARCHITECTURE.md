# WitDatabase Architecture

## Executive Summary

WitDatabase is an embedded key-value database engine for .NET with two storage backends:
- **B+Tree**: Traditional disk-based storage with ACID transactions
- **LSM-Tree**: Log-structured merge-tree optimized for write-heavy workloads

**Target Platforms**: .NET 9, .NET 10  
**Total Tests**: 1026  
**Test Coverage**: ~85% of core functionality

## Known Issues & Limitations

### Critical

None currently known.

### High Priority

1. **Single-writer transactions**: No concurrent writers
2. **FileLock reliability**: May fail on network filesystems
3. **No compression**: SSTables are uncompressed

### Medium Priority

1. **No read-only transactions**: All reads acquire locks
2. **No savepoints**: Cannot partially rollback
3. **Large value handling in LSM**: No separate blob storage
4. **No online compaction tuning**: Fixed trigger threshold

### Low Priority

1. **No secondary indexes**: Key-only access
2. **No TTL support**: No automatic expiration
3. **No replication**: Single-node only
4. **Limited monitoring**: Basic statistics only

---

## Recommendations

### For Production Use

1. **Use LSM-Tree for write-heavy workloads**
   - Enable WAL (`EnableWal = true`)
   - Set appropriate `MemTableSizeLimit`
   - Enable block cache for read performance

2. **Use B+Tree for read-heavy workloads**
   - Configure adequate page cache size
   - Use `FileStorage` with `WriteThrough` for durability

3. **Enable encryption for sensitive data**
   - Use secure key management
   - Consider HSM integration via custom `ICryptoProvider`

### For Transactional Use

1. **Be aware of single-writer limitation**
2. **Keep transactions short**
3. **Always dispose transactions** (auto-rollback on dispose)
4. **Call `Checkpoint()` periodically** to truncate WAL

### For High Performance

1. **Tune cache sizes** based on working set
2. **Use async APIs** in async contexts
3. **Batch operations** where possible (manual batching)
4. **Monitor statistics** (`LsmStatistics`)
